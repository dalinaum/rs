# 만들기 전에 확인하기

## 무슨 일이 있었나

PR-4 "RS 분포 히스토그램 차트 추가"는 실제로 필요하지 않은 기능이었다. 사용자가 "2, 3, 4를 같이 다른 worker 에이전트로 병렬 진행해봐"라고 지시했고, AI 에이전트가 PR-2, PR-3, PR-4를 동시에 만들었다. PR-4의 결과물을 나중에 확인해보니 쓸모가 없었다.

## 왜 이런 일이 생겼나

1. **계획 단계에서 각 작업의 필요성을 충분히 논의하지 않았다.** AI가 개선 아이디어 목록을 제안했고, 사용자가 전체 목록을 한 번에 승인하는 형태로 진행되었다. 개별 항목에 대해 "이게 정말 필요한가?"를 묻거나 답하는 과정이 없었다.

2. **병렬 실행이 검증 기회를 없앴다.** 순차적으로 진행했다면 PR-4를 시작하기 전에 "이 히스토그램이 실제로 유용한가?"를 생각할 시간이 있었을 것이다. 병렬로 한꺼번에 던지면서 그 기회가 사라졌다.

3. **결과물을 보기 전까지 판단이 어려웠다.** "RS 분포 히스토그램"이라는 설명만으로는 유용할 것 같았지만, 실제 차트를 보고 나서야 불필요하다는 것을 알 수 있었다.

## 앞으로 어떻게 할 것인가

### 사용자 (나)
- 여러 작업을 한 번에 지시하기 전에, 각 항목이 정말 필요한지 하나씩 생각한다.
- AI가 제안한 목록을 통째로 승인하지 않는다. "이건 왜 필요하지?"를 항목마다 묻는다.
- 확신이 없는 기능은 병렬 작업에서 빼고, 먼저 확실한 것부터 진행한다.

### AI 에이전트 (Claude)
- 기능을 제안할 때 "이것이 사용자에게 실제로 가치가 있는가?"를 스스로 점검한다.
- 여러 작업을 병렬로 진행하기 전에, 각 작업의 필요성을 사용자에게 개별적으로 확인한다. "PR-4로 RS 분포 히스토그램을 추가하려 합니다. 이 기능이 필요하신가요?"처럼 묻는다.
- 확실한 작업과 불확실한 작업을 구분하여, 불확실한 것은 프로토타입이나 설명을 먼저 보여준 뒤 진행 여부를 결정한다.

## 핵심 원칙

**"만들 수 있는가"보다 "만들어야 하는가"를 먼저 확인한다.**

속도를 위해 병렬로 진행하는 것은 좋지만, 병렬로 진행할 항목 각각이 정말 필요한지를 먼저 합의해야 한다. 불필요한 기능을 빠르게 만드는 것은 낭비다.
